# PostCraft 产品需求文档（PRD）

> **版本**：v0.2  
> **更新日期**：2026-07-01  
> **文档状态**：初稿  
> **产品代号**：PostCraft  
> **产品化名称（推荐）**：生活有稿  
> **产品定位**：个人观察型内容创作工作台 / 从灵感到发布的 AI 内容工作室

---

## 1. 产品概述

### 1.1 一句话定位

**把生活观察与碎片灵感，整理成可发布的多平台内容（公众号 / 小红书 / 抖音等），并通过对话式协作不断打磨至可发布状态。**

### 1.2 产品要解决的核心问题

用户不缺想法，缺的是从「灵感」到「可发布内容」之间的完整链路：

| 痛点 | 描述 |
| --- | --- |
| 灵感易流失 | 走路、刷手机时的感触没有统一入口沉淀 |
| 中间环节繁琐 | 搭结构、补论据、润色、改平台风格、想标题、做封面，步骤多且重复 |
| 一稿难多发 | 同一选题在不同平台需要完全不同的表达方式和排版 |
| 反复修改成本高 | 生成后不满意，需要多轮调整，缺乏「边聊边改」的协作体验 |
| 发布前缺乏预览 | 看不清最终在小红书/公众号/抖音上的呈现效果 |

### 1.3 产品价值主张

- **不是**单纯的 AI 写作工具、自媒体矩阵工具或爆款文案生成器  
- **而是**覆盖「灵感 → 资料整理 → 内容生成 → 多平台改写 → 配图/封面 → 发布记录 → 数据复盘」的 **个人观察型内容创作工作台**  
- **设计优先级**：先帮用户把想法变成高质量、有个人风格的内容，再辅助发布与复盘（而非追求一键分发）  
- **核心差异化**：对话式内容打磨 + 多平台预览 + 一稿多发 + 真实表达（去 AI 味 / 去营销味）

### 1.4 目标用户

- **用户类型**：个人创作者，同时在公众号、小红书、抖音等平台发布内容  
- **内容类型**：**观察型个人创作**——把生活观察系统化变成可发布内容，而非泛营销文案或矩阵批量内容  
- **内容气质**：真实、观察、生活化、有观点、有温度、有分寸（非营销号 / 非爆款套路风格）  
- **典型选题**：社会观察、农村生活、健康消费、普通人生活记录、消费陷阱、环境污染等  

**内容支柱（Content Pillars）示例**——用于选题库标签、系列化与复盘分组：

| 支柱 | 示例选题 |
| --- | --- |
| 农村老人与家庭健康 | 农村老人重疾增多、就医意识、家庭照护 |
| 消费陷阱与三无产品 | 劣质商品、虚假宣传、普通家庭避坑 |
| 农村环境与普通人风险 | 污染观察、生活安全、日常健康隐患 |

### 1.6 市场定位与竞品空白

市场上已有大量内容工具，但多数只能做到「生成内容 + 辅助排期/发布」，尚未有产品 **同时** 较好地覆盖 **公众号深度长文 + 小红书共鸣笔记 + 抖音口播脚本**，且真正适合 **个人深度观察型创作**。

**竞品大致分三类**：

| 类型 | 代表 | 擅长 | 不足（相对 PostCraft） |
| --- | --- | --- | --- |
| **AI 内容生成工具** | ChatGPT、Claude、豆包、Kimi、秘塔写作猫、Notion AI | 写稿、改稿、起标题、扩选题 | 不负责发布；弱多平台适配与预览；不沉淀选题资产与创作复盘 |
| **多平台发布/排期工具** | Buffer、Hootsuite、Agorapulse、Postudio 等 | 排期、分发、矩阵管理、部分 AI 辅助 | 偏发布效率，不懂个人风格；国内平台（小红书/抖音/公众号）支持不稳定或仍在 Beta |
| **AI 自媒体/矩阵运营工具** | KeduckAI、各类「一键分发」方案 | 批量生成、多账号、自动发布、热点追踪 | 内容易模板化；适合营销矩阵，不适合个人表达；自动发布有账号风控风险 |

**PostCraft 占位**：个人观察型创作者的内容工作台——**先创作质量，后发布效率**。

**现成平台 vs PostCraft（摘要）**：

| 方向 | 现成平台 | PostCraft |
| --- | --- | --- |
| 内容生成 | 快，但易模板化 | 对话迭代 + humanizer，贴近个人表达 |
| 多平台发布 | 部分支持，但不稳定 | MVP 手动复制发布 + 发布记录 |
| 小红书/抖音适配 | 偏爆款套路 | 更生活化、真实、有分寸 |
| 公众号深度内容 | 普遍一般 | 重点优化长文结构与观点表达 |
| 选题资产沉淀 | 多数较弱 | 灵感库 → 选题库 → 系列支柱 |
| 数据复盘 | 偏运营大盘 | 结合创作方向与选题体系分析 |
| 风格一致性 | 不稳定 | 作者风格档案 + 长期表达沉淀（v1.1 起） |

### 1.7 我们刻意不做的事（反定位）

以下能力 **刻意不在 MVP 追求**，避免产品滑向同质化「分发工具」：

1. **不做自媒体矩阵 / 批量发内容工具**——目标用户是个人创作者，不是 MCN 运营  
2. **不做爆款标题 / 营销笔记生成器**——默认语气为观察、共情、分寸，而非「震惊」「必看」  
3. **不做第一版一键自动发布**——平台 API 限制多、接口不稳定，自动发布易触发账号风控；MVP 以复制导出 + 发布记录为主  

### 1.8 第一版发布策略

MVP **主动选择**手动发布路径，将研发重心放在「创作质量」验证上：

| 考量 | 说明 |
| --- | --- |
| 平台接口 | 小红书、抖音、微信公众号官方 API 限制多，第三方方案不稳定 |
| 账号安全 | 自动发布、批量操作易触发平台风控 |
| 产品验证 | 第一版需验证的是「从灵感到满意成稿」，而非「分发效率」 |
| MVP 替代方案 | 各平台一键复制 + **发布记录**（平台 / 时间 / 链接 / 状态） |
| 后续版本 | v1.1 发布日历；更晚版本再评估平台 API 接入 |

### 1.9 内容表达原则（分寸感）

观察型内容最需要 **真实感、观点、共情、分寸感**，产品默认遵循以下原则（贯穿生成、对话修改与风险扫描）：

- 默认 **「现象观察」** 语气，避免绝对化、定论式批判  
- 健康/医疗相关话题自动提示 **免责声明**（个人观察，非医疗建议）  
- 对话快捷指令支持：「再温和一点」「去掉说教感」「加个人经历、少堆数据」「检查敏感表述」  
- 与 `humanizer-cn` 配合：**去 AI 味 + 去营销味 + 保留人味**

## 2. 产品目标与范围

### 2.1 MVP 目标（第一版）

在 **4–6 周** 内交付可用版本，验证核心价值：

1. 用户输入灵感，系统生成多平台内容初稿  
2. 用户通过 **对话聊天框** 持续修改文案、排版、配图  
3. 用户可在 **预览面板** 查看各平台近似最终效果  
4. 内容可保存为草稿，支持复制导出，**手动发布**并记录发布信息（不接平台 API）

### 2.2 MVP 范围（In Scope）

| 模块 | MVP 是否包含 |
| --- | --- |
| 灵感输入 | ✅ |
| 选题扩展与结构生成 | ✅ |
| 多平台文案生成（公众号 / 小红书 / 抖音） | ✅ |
| 对话式内容迭代 | ✅ **核心** |
| 多平台内容预览 | ✅ **核心** |
| 标题批量生成 | ✅ |
| 封面文案 + 配图提示词 + AI 生图 | ✅ |
| 排版/格式化输出 | ✅ |
| 草稿保存与管理 | ✅ |
| 发布记录（平台 / 时间 / 链接） | ✅ **轻量** |
| 素材/事实核查提示 | ⚠️ 基础版（风险提示） |
| 作者风格档案 | ⚠️ P1（基础语气与禁用词） |
| 发布日历 | ❌ v1.1 |
| 数据复盘 | ❌ v1.1 |
| 平台 API 自动发布 | ❌ 后续版本（见 §1.8） |
| 语音输入 / 链接抓取 | ❌ 后续版本 |

### 2.3 非目标（Out of Scope — MVP）

- 多用户协作、团队权限  
- 平台账号绑定与一键自动发布（见 §1.7、§1.8）  
- 自媒体矩阵、批量内容生成与营销号套路  
- 复杂数据分析与自动选题推荐  
- 移动端独立 App（MVP 以 Web 为主）

## 3. 核心用户流程

### 3.1 主流程（Happy Path）

```text
输入灵感
  → 系统生成选题角度 + 文章结构
  → 一键生成多平台初稿（公众号 / 小红书 / 抖音）
  → 右侧/底部预览各平台效果
  → 在对话区提出修改（「标题再犀利一点」「小红书版加 emoji」「封面要暖色乡村风」）
  → AI 理解上下文，更新对应平台内容
  → 预览实时刷新
  → 满意后保存草稿 / 复制 / 导出
  → 手动发布到各平台，填写发布记录（平台 / 时间 / 链接）
```

### 3.2 典型使用场景

**场景 A：一句话灵感成稿**

> 输入：「农村老人五六十岁重疾增多，可能和劣质商品、环境污染、医疗意识有关」  
> 输出：公众号深度文 + 小红书共鸣笔记 + 抖音 60 秒口播脚本 + 20 个标题 + 封面提示词

**场景 B：对话式精修**

> 用户：「公众号开头太说教，改成从回农村的经历切入」  
> 系统：仅更新公众号版本开头，保留其他平台内容  
> 用户：「小红书版段落太长，拆成 5 段，每段加小标题」  
> 系统：更新小红书排版结构，预览区同步变化

**场景 C：配图与封面**

> 用户：「生成 3 张小红书封面，风格要真实纪实、暖色、不要 AI 感太重」  
> 系统：输出封面文案 + 生图提示词 + 可选 AI 生图结果，嵌入预览

### 4.3 灵感库

**目标**：不让灵感流失，为后续选题提供原料。

| 功能 | 描述 | MVP |
| --- | --- | --- |
| 文本灵感 | 一句话、段落、手动笔记 | ✅ |

### 4.5 内容创作室（核心模块）

这是 MVP 的 **核心页面**，采用 **三栏布局**（可响应式折叠）：

```text
+-------------------------------------------------------------+
|  顶栏：项目标题 | 平台切换 | 保存 | 导出 | 返回                |
+--------------+--------------------------+-------------------+
|              |                          |                   |
|  对话区      |   内容编辑 / 结构化视图    |   多平台预览区     |
|  (Chat)      |   (Generated Content)    |   (Preview)       |
|              |                          |                   |
|  用户与 AI   |   当前平台文案             |   公众号排版预览   |
|  多轮对话    |   标题列表                 |   小红书卡片预览   |
|  修改指令    |   口播脚本分镜             |   抖音脚本预览     |
|              |   封面/配图区              |   封面图预览       |
|              |                          |                   |
+--------------+--------------------------+-------------------+
```

#### 4.5.1 多平台文案生成

**输入**：

- 灵感/选题描述  
- 可选：内容方向、情绪基调、目标平台、目标字数  

**输出 — 按平台差异化生成**：

| 平台 | 内容特点 | 默认结构 | 字数参考 |
| --- | --- | --- | --- |
| **微信公众号** | 深度、情绪、观点、故事化 | 共鸣开头 → 现象 → 原因分析 → 案例 → 建议 → 升华结尾 | 1500–3000 字 |
| **小红书** | 标题抓人、短段落、真实感、互动 | 痛点标题 → 生活化开头 → 分段要点 → 个人感受 → 提醒 → 互动提问 | 500–1000 字 |
| **抖音** | 口播、短句、强节奏 | 3 秒钩子 → 冲突 → 3 个原因 → 情绪推进 → 金句 → 引导评论 | 60s / 90s / 3min 可选 |
| **视频号** | 克制、熟人社交感 | 类似抖音但更短更稳 | v1.1 |
| **知乎** | 理性、重逻辑论据 | 问题定义 → 分论点 → 论据 → 结论 | v1.1 |

**功能要求**：

- 支持单平台或一键生成全部已选平台  
- 各平台内容 **独立存储**，互不影响  
- 生成时可选择「只重新生成当前平台」  
- 生成结果结构化展示（标题 / 正文 / 标签 / 口播分镜）

#### 4.5.2 对话式内容迭代（核心）

**目标**：用户像和编辑聊天一样，逐步完善内容，而不是一次性生成后手动大改。

| 需求 ID | 描述 | 优先级 |
| --- | --- | --- |
| CHAT-01 | 提供持久化对话线程，绑定到单个内容项目 | P0 |
| CHAT-02 | AI 理解当前项目的完整上下文（灵感、各平台已有文案、用户历史修改） | P0 |
| CHAT-03 | 支持自然语言修改指令，如「标题更共鸣」「小红书缩短」「抖音加钩子」 | P0 |
| CHAT-04 | 支持 **指定平台** 修改：「只改公众号版」「小红书和其他平台都改」 | P0 |
| CHAT-05 | 每次 AI 回复需明确 **变更摘要**（改了什么、为什么） | P0 |
| CHAT-06 | 支持 **撤销/回退** 到上一轮内容版本 | P1 |
| CHAT-07 | 支持快捷指令按钮：「更温和」「更犀利」「缩短 30%」「加案例」「检查敏感表述」 | P1 |
| CHAT-08 | 对话中可引用预览区选中段落：「把这段改得更生活化」 | P1 |
| CHAT-09 | 流式输出（Streaming），减少等待焦虑 | P1 |

**对话示例**：

```text
用户：基于这个选题，先生成三个平台的初稿
AI：  已生成公众号/小红书/抖音初稿，请在预览区查看。公众号侧重深度分析…

用户：公众号开头太硬，改成我回农村过年的亲身经历
AI：  已更新公众号版开头（其他平台未改动）→ 预览已刷新

用户：小红书标题太平，给我 10 个更抓人的备选
AI：  已生成 10 个标题，已追加到标题列表

用户：用第 3 个标题，并把正文改成匹配这个标题的角度
AI：  已更新小红书标题与正文结构
```

#### 4.5.3 多平台预览（核心）

**目标**：用户在发布前，近似看到内容在各平台的呈现效果。

| 需求 ID | 描述 | 优先级 |
| --- | --- | --- |
| PREV-01 | 平台 Tab 切换：公众号 / 小红书 / 抖音 | P0 |
| PREV-02 | **公众号预览**：标题 + 摘要 + 正文排版（段落、小标题、引用） | P0 |
| PREV-03 | **小红书预览**：封面图 + 标题 + 正文卡片式排版 + 标签 + emoji 风格 | P0 |
| PREV-04 | **抖音预览**：口播脚本分镜表（序号 / 文案 / 时长 / 画面建议） | P0 |
| PREV-05 | 预览内容与编辑区 **实时同步** | P0 |
| PREV-06 | 封面图/配图在预览中占位展示 | P0 |
| PREV-07 | 支持「预览模式 / 编辑模式」切换 | P1 |
| PREV-08 | 支持复制当前平台全文（含排版 Markdown / 纯文本） | P0 |
| PREV-09 | 支持导出为 Markdown / 纯文本 / 口播脚本表格 | P1 |

#### 4.5.4 排版与格式化

| 平台 | 排版要求 |
| --- | --- |
| 公众号 | 段间距、小标题层级、引用块、重点加粗、适合复制的 HTML/Markdown |

| 需求 ID | 描述 | 优先级 |
| --- | --- | --- |
| FMT-01 | 各平台独立排版引擎（规则 + AI） | P0 |
| FMT-02 | 用户可通过对话调整排版：「小红书每段不超过 2 行」 | P0 |

#### 4.5.5 标题生成

| 需求 | 描述 |
| --- | --- |

#### 4.5.7 素材与事实核查（基础版）

| 检查项 | 动作 |

### 4.9 作者风格档案（P1 → v1.1 增强）

**目标**：让生成与修改内容长期贴近 **个人表达方式**，区别于模板化矩阵工具。

| 能力 | MVP（P1 基础） | v1.1 增强 |
| --- | --- | --- |
| 语气偏好 | 温和观察 / 不煽情 / 理性共情 | 多套风格预设切换 |
| 禁用表达 | 如「震惊」「必看」「赶紧转发」 | 用户自定义黑名单 |
| 个人素材片段 | 可复用的经历、案例（如回农村见闻） | 创作时 AI 主动引用 |
| 默认篇幅与结构 | 公众号偏故事开头、小红书偏短段落 | 按平台记忆偏好 |
| 注入生成流水线 | 写入 Skill 上下文（general-writing / humanizer-cn） | 全链路风格一致性校验 |

**与竞品差异**：矩阵工具追求「快和像爆款」；PostCraft 追求「慢打磨、像本人、可长期沉淀」。

## 8. 技术方案建议（供研发参考）

### 8.1 推荐技术栈

| 层级 | 建议 |
| --- | --- |
| 前端 | Next.js + React + Tailwind CSS |
| 后端 | Python FastAPI |
| 数据库 | SQLite（本地 MVP）→ PostgreSQL（云端） |
| AI 对话 | DeepSeek / OpenAI（`DEEPSEEK_API_KEY` / `OPENAI_API_KEY`） |
| 生图 | 通义万相 / DALL-E / Stable Diffusion API |
| 状态管理 | Zustand / React Query |

### 8.2 关键架构：对话 + 预览联动

```text
用户消息
  → Chat Orchestrator（解析意图：改哪个平台、改什么）
  → Context Builder（注入当前各平台文案 + 选题 + 历史）
  → LLM 生成结构化 Patch（非全文重写优先）
  → Content Store 更新对应平台字段
  → Preview Renderer 重新渲染
  → 对话区返回变更摘要
```

**设计原则**：

1. **结构化存储**：各平台内容分开存，对话修改时精准 patch  
2. **Preview Driven**：预览区是「验收标准」，对话修改以预览效果为准  
3. **版本可追溯**：重大修改留快照，支持回退  

详细设计见 [docs/architecture/chat-orchestrator.md](./architecture/chat-orchestrator.md)。

### 8.3 Skill 驱动架构

PostCraft 的内容生成不过度硬编码 Prompt，而是基于 [oh-my-writing-skill](https://github.com/z0gSh1u/oh-my-writing-skill/tree/master/skills) + PostCraft 自研 Skill 链：

```text
Chat Orchestrator
  → SkillLoader（读取 SKILL.md）
  → Pipeline（general-writing → humanizer-cn → converters）
  → ContentPatch → Preview
```

Skill 目录说明见 [skills/README.md](../skills/README.md)。

## 12. 风险与对策

| 风险 | 对策 |
| --- | --- |
| AI 生成内容过于「AI 味」 | 内置「真实表达」系统 Prompt + humanizer-cn + 对话快捷风格调整 |
| 内容模板化 / 爆款套路感 | 反定位约束 + 作者风格档案 + 默认观察型语气（见 §1.7、§1.9） |
| 健康/社会话题表述风险 | 事实核查基础模块 + 默认温和观察语气 + 免责声明提示 |
| 平台 API 不稳定 / 账号风控 | MVP 不做自动发布；手动复制 + 发布记录（见 §1.8） |
| 多平台预览与真实平台有差异 | 标注「近似预览」；提供复制后微调空间 |
| 生图质量不稳定 | 提示词模板 + 多候选 + 手动上传兜底 |
| 对话上下文过长 | 摘要压缩 + 结构化上下文注入，非全文堆叠 |

## 15. 附录

### 15.1 平台内容风格 Prompt 要点（摘要）

**公众号**：故事化开头、有观点、段落长但有节奏、结尾升华  
**小红书**：第一人称、短段落、真实细节、轻 emoji、强互动结尾  
**抖音**：口语化、短句、强钩子、三段论、金句收尾  

### 15.2 参考竞品与差异化

详见 **§1.6 市场定位与竞品空白**。摘要如下：

| 竞品类型 | 代表 | PostCraft 差异 |
| --- | --- | --- |
| 通用 AI 写作 | ChatGPT、秘塔、Notion AI | 多平台结构化生成 + 对话 patch + 平台预览 + 选题资产 |
| 多平台发布工具 | Buffer、Postudio、Agorapulse | 不做 MVP 一键发布；专注创作质量与个人风格 |
| AI 矩阵运营 | KeduckAI、一键分发方案 | 非批量营销；观察型表达 + 分寸感 + 作者风格沉淀 |
| 排版/单平台工具 | 各类公众号排版器 | 从灵感到选题到三平台成稿 + 配图一体化 |

**核心差异一句话**：帮个人创作者把 **生活观察** 系统化变成公众号、小红书、抖音内容——**先成好稿，再辅助发布**。
